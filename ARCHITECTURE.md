# 地牢冒险小游戏 — 架构说明

> 面向维护者的技术文档。本文档描述代码组织结构、各模块职责、核心流程和算法思路，方便后续接手修改。

---

## 1. 整体概览

这是一个使用 Go 编写的终端地牢冒险 Roguelike 游戏。全程在终端（Windows PowerShell / cmd）中运行，无外部依赖，仅使用 Go 标准库。玩家通过 WASD 键控制角色 `@` 在随机生成的地牢中探索、战斗、捡取道具，共 5 层，第 5 层有 Boss。

### 核心设计原则

- **纯标准库**：不引入任何第三方 Go 包，`go build .` 即可编译。
- **同包多文件**：所有代码在 `package main` 下，按职责拆分到 5 个 `.go` 文件，彼此通过同包名直接访问，没有循环依赖。
- **单线程顺序执行**：没有 goroutine、没有 channel，所有逻辑按「输入 → 更新状态 → 渲染」的循环顺序执行，便于调试。
- **硬删除优于软删除**：怪物击杀后直接从切片移除，而非保留 HP ≤ 0 的死怪。这样遍历时不需要额外的 HP 守卫，从根本上避免了「遗漏检查」类 Bug。

## 2. 代码结构与文件职责

```
project38/
├── main.go        # 主循环 + 键盘输入（入口文件，只留流程骨架）
├── entity.go      # 所有数据结构、常量、构造函数（头文件）
├── mapgen.go      # 地图生成：房间、走廊、实体放置、视野、换层
├── battle.go      # 战斗系统：移动、战斗开始/攻击/逃跑、工具函数
├── render.go      # 渲染：地图/战斗/结算画面、清屏、Windows ANSI 初始化
└── dungeon.exe    # 编译产物（可执行）
```

### 2.1 entity.go — 数据定义层

这是整个项目的「头文件」，所有其他文件都依赖它。包含：

- **常量**：地图尺寸（60×20）、最大层数（5）、视野半径（5 格）、Tile 字符定义（`#`/`.`/`@`/`M`/`+`/`$`/`>`/`B`）、ANSI 颜色转义序列
- **数据结构体**：
  - `Player`：坐标、HP/MaxHP、ATK、DEF、Gold、当前层数/最深层数
  - `Monster`：坐标、HP/MaxHP、ATK、DEF、Gold、名字、是否 Boss
  - `Game`：顶层状态容器，持有地图瓦片、Visited 迷雾数组、玩家、怪物切片、药水/金币 HashMap、出口坐标、消息、战斗状态、游戏结束标志
  - `Battle`：战斗状态，持有对战怪物指针 + 战斗日志切片
  - `Room`：矩形房间（X/Y/W/H），带几何计算方法
- **构造函数**：`NewPlayer()` 创建初始属性角色（HP30/ATK5/DEF2）
- **Room 方法**：中心点计算、矩形相交检测（房间生成时用）

### 2.2 mapgen.go — 地图生成层

负责「世界」的生成和更新，与战斗/渲染完全解耦：

- `GenerateMap()`：单层地图的总入口，依次执行「初始化二维数组 → 生成房间 → 连接走廊 → 放置玩家出生点 → 放置出口 → 放置实体 → 刷新视野」
- `generateRooms()`：**随机房间生成算法**（见第 4 节详述）
- `connectRooms()` + `carveHCorridor/VCorridor`：**走廊连接算法**（见第 4 节详述）
- `PlaceEntities()`：按层数随机散布怪物、药水、金币；第 5 层额外放置 Boss
- `spawnMonster()` / `placeBoss()` / `placeRandomItem()`：实体放置的具体实现，均带 50 次重试和合法性检测
- `isSpawnable()` / `hasEntity()`：放置合法性检测（底层瓦片必须是地板 + 无其他实体占位）
- **视野系统**：
  - `UpdateVisibility()`：以玩家为中心画圆（欧氏距离 ≤ 5²），将圆内格子标记为已探索
  - `InVision()`：判断某格是否在当前视野内（圆内）
- `NextFloor()`：层数自增、刷新最深层记录、重新生成整张地图

### 2.3 battle.go — 战斗 & 移动层

处理所有「玩家主动操作导致状态改变」的逻辑：

- `MovePlayer(dx, dy)`：玩家移动总入口
  - 边界/墙壁检测
  - 撞怪检测：命中活怪则开始战斗，不移动
  - 合法移动：更新坐标、刷新迷雾视野
  - 触发格子效果：药水回血、金币拾取、到达出口则换层
- `startBattle(m)`：初始化战斗状态，填入战斗首条日志
- `removeMonster(target)`：**核心方法** — 通过指针比对从切片中安全移除怪物（硬删除），这是保证系统一致性的关键
- `BattleAttack()`：攻击流程
  - 计算玩家伤害（`max(1, ATK-DEF)`）
  - 怪死 → 发奖励金币、从切片移除怪物、结束战斗
  - 怪活 → 怪物反击、可能触发玩家死亡
- `BattleFlee()`：逃跑流程
  - Boss 战不可逃，必遭反击
  - 普通怪 50% 成功概率，成功就结束战斗，失败则挨一下打
- `max(a, b)` / `min(a, b)`：通用整数工具

### 2.4 render.go — 渲染层

纯输出逻辑，**不会修改 Game 状态**：

- `init()` 包初始化：注册 Windows/Linux 的清屏函数、启用 Windows 虚拟终端 ANSI 转义支持
- `enableWindowsANSI()`：调用 kernel32.dll 的 `SetConsoleMode` 开启 ENABLE_VIRTUAL_TERMINAL_PROCESSING 标志，确保 Win10+ 终端能正确解析 ANSI 颜色
- `ClearScreen()`：跨平台清屏
- `Render()`：三态迷雾渲染（核心见第 5 节）
  - 构建 screen 副本（因为底层 Map 瓦片会被实体符覆盖）
  - 逐行逐字符判定状态：未探索 = 空格；已探索但离开视野 = 深灰色地形（不显示动态实体）；当前视野内 = 正常彩色
  - 底部状态栏：HP/ATK/DEF/金币/当前层数
- `RenderBattle()`：战斗界面，展示双方属性条 + 滚动日志
- `RenderGameOver()`：结算界面，胜利/死亡画面 + 最终得分 + 最深到达层数

### 2.5 main.go — 入口 & 主循环

整个文件只有两个函数，尽量薄：

- `readKey()`：从 stdin 读取 1 字节按键（原始模式由终端外部保证）
- `main()`：
  - 播种随机数
  - 外层循环：创建新 Game → 生成地图 → 渲染
  - 内层循环：读按键 → 分发到对应处理函数 → 重新渲染
  - 游戏结束后按 R 重开，其他键退出

## 3. 游戏主循环流程

```
┌──────────────────────────────────────────────┐
│           main() 外层循环（重开）            │
│                                              │
│   game = &Game{Player: NewPlayer()}         │
│   game.GenerateMap()    ← 房间/走廊/实体    │
│   game.Render()         ← 初始画面          │
│                                              │
│   ┌──────────────────────────────────────┐   │
│   │        内层循环（回合制）            │   │
│   │                                      │   │
│   │   c = readKey()                      │   │
│   │                                      │   │
│   │   if GameOver:                       │   │
│   │     R → break 内层, 重开游戏        │   │
│   │     其他 → os.Exit(0)               │   │
│   │                                      │   │
│   │   if in Battle:                      │   │
│   │     A → BattleAttack()              │   │
│   │     F → BattleFlee()                │   │
│   │                                      │   │
│   │   else (地图模式):                   │   │
│   │     W → MovePlayer(0, -1)           │   │
│   │     A → MovePlayer(-1, 0)           │   │
│   │     S → MovePlayer(0, 1)            │   │
│   │     D → MovePlayer(1, 0)            │   │
│   │                                      │   │
│   │   game.Render()  ← 每帧重绘         │   │
│   └──────────────────────────────────────┘   │
│                                              │
└──────────────────────────────────────────────┘
```

**关键时序约定：**

1. **一回合 = 一次按键 + 一次渲染**。没有独立的时间流逝，怪物完全是「被动」的，只有玩家撞到它们才触发交互（这是简化版设计，如要实现怪物 AI 巡逻，需要在 MovePlayer 末尾加怪物 AI tick）。
2. **战斗是模态的**：`g.Battle != nil` 时 WASD 键被完全忽略，只有 A/F 有效。这避免了战斗中「又移动又攻击」的状态混乱。
3. **渲染与逻辑严格分离**：`Render*` 系列方法只读不写，所有状态变更集中在 `MovePlayer` / `BattleAttack` / `BattleFlee` 里，调试时只要打印 Game 结构体就能复现问题。

## 4. 地图生成算法

### 4.1 房间生成（Random Room Placement with Separation）

目标是生成 6–9 个不重叠的矩形房间，随机分布在 60×20 的地图上。

**算法流程：**

1. 初始化目标房间数：`6 + rand.Intn(4)`（即 6–9 间）
2. 最多尝试 100 次（`maxAttempts`），直到达到目标房间数：
   - 随机生成房间尺寸：宽 4–10，高 3–7
   - 随机生成房间左上坐标：保证四周留 1 格墙壁边界
   - **相交检测**：对每个已存在的房间，用 `Room.Intersects()` 判断是否重叠。Intersects 采用「扩展 1 格」的策略——两个房间之间强制留 1 格墙的间隔，避免房间直接相连。这也是为什么虽然每层有 6–9 个房间，但玩家在视觉上感觉它们被走廊明确分隔。
   - 不相交 → 加入房间列表，并把矩形区域的瓦片从墙 `#` 改成地板 `.`
3. 尝试结束。即使达不到目标数量（比如地图太挤），也正常返回已有房间，保证游戏能继续。

**复杂度**：O(maxAttempts × n)，n 为已生成房间数。对于 100×10 的规模完全可忽略。

### 4.2 走廊连接（L-shaped Corridors）

目标是确保所有房间两两连通（实际采用「链式连通」）。

**算法流程：**

1. 按房间生成顺序，依次连接第 i–1 间和第 i 间
2. 取两房间的中心点 `(ax, ay)` 和 `(bx, by)`
3. **随机选择 L 形走向**：
   - 50% 概率：先水平挖一段，再垂直挖一段（先在 `ay` 行从 `ax` 挖到 `bx`，再在 `bx` 列从 `ay` 挖到 `by`）
   - 50% 概率：先垂直挖一段，再水平挖一段
4. 走廊宽度固定为 1 格，挖的时候将瓦片改成地板

**设计考量：**

- 「链式连通」比「全连通」或「最小生成树」简单，但仍能保证整张地图连通——因为 1→2→3→…→n 天然构成一条通路。
- 随机 L 形走向避免了所有走廊都一模一样的视觉重复感。
- 走廊可能与其他房间/走廊相交，这是**期望的**——会形成自然的交叉路口和「大房间」。

### 4.3 实体放置

玩家出生在**第一个房间中心**，出口放在**最后一个房间中心**（第 5 层出口被 Boss 覆盖，改为击杀 Boss 即通关）。其他实体放置规则：

- **放置合法性**：底层瓦片必须是地板 + 没有其他实体（玩家/怪/药/金币/出口）占位
- **重试机制**：每次放置最多重试 50 次，找不到合法位置就跳过这个实体（不至于卡死）
- **怪物**：从**第二个房间起**随机选房 → 房内随机取点 → 合法就放；属性随层数线性增长
- **药水**：与怪物相同放置策略；回血量 = 8–15 + 层数×2
- **金币**：相同策略；金额 = 5–14 + 层数×3
- **Boss**：第 5 层强制放在最后一个房间中心，覆盖出口，血量 60 ATK12 DEF5，击杀后触发胜利结算

## 5. 战争迷雾系统

采用经典的「当前视野 + 地形记忆」双轨模型：

### 5.1 数据结构

`Game.Visited [][]bool`：二维布尔数组，大小等于地图，记录某格是否**曾经**被玩家看到过。一旦标为 `true` 永不重置（同一层内），即永久地形记忆。

### 5.2 视野判定

采用**欧氏距离圆形视野**，半径 = 5：
```
InVision(x, y) = (x - px)² + (y - py)² ≤ 5²
```
为什么不用曼哈顿或切比雪夫？圆形在终端视觉上更自然，不会出现明显的方形边界感。

每次玩家移动后调用 `UpdateVisibility()`，遍历 11×11 = 121 格的方形 bounding box，把圆内格标为 `Visited = true`。

### 5.3 三态渲染

渲染时对每一格做分支判断：

| 状态 | 条件 | 显示效果 |
|------|------|---------|
| **当前视野内** | `InVision(x, y) == true` | 正常显示所有内容：地形 + 动态实体（怪/药/金币/Boss/出口/玩家） |
| **已探索，不在视野** | `Visited == true && !InVision` | 深灰色（ANSI `\x1b[90m`）显示**仅地形**；动态实体一律用底层瓦片代替，即：离开视野后你看不到怪物和道具是否还在，符合「战争迷雾」的探索感 |
| **未探索** | `Visited == false` | 直接显示空格，完全看不到 |

**实现注意**：逐行渲染时维护 `prevGray` 标志，只在灰/彩色切换时才插入 ANSI 转义码，避免每行每格都插转义序列导致终端吞吐压力。行结束时强制 reset 颜色，防止状态栏被染色。

## 6. 战斗系统

### 6.1 状态切换

```
地图模式 (Battle == nil)
    │  走到怪物格子
    ▼
战斗模式 (Battle != nil)
    │  怪物HP ≤ 0 或 逃跑成功
    ▼
地图模式

战斗模式
    │  玩家HP ≤ 0
    ▼
GameOver = true (死亡结算)

战斗模式 (Boss)
    │  Boss HP ≤ 0
    ▼
GameOver = true && Win = true (胜利结算)
```

### 6.2 伤害公式

```
实际伤害 = max(1, 攻击方ATK - 防御方DEF)
```

永远至少造成 1 点伤害，避免「攻防相抵打不动」的死循环。

### 6.3 逃跑规则

- Boss 战：100% 无法逃跑，且必定遭受反击
- 普通怪：50% 成功（`rand.Intn(2) == 0`），失败则挨一下打

## 7. 常见维护场景指引

### 7.1 调整难度

- 玩家初始属性：`entity.go` 中 `NewPlayer()`
- 怪物数量/属性随层增长：`mapgen.go` 中 `PlaceEntities()` 和 `spawnMonster()` 的 `hp/atk/def` 计算
- 伤害公式：`battle.go` 中 `BattleAttack()` / `BattleFlee()` 的 `max(1, ...)` 行
- 视野大小：`entity.go` 中 `VisionRadius` 常量

### 7.2 新增道具类型

1. 在 `entity.go` 中加新 Tile 常量（如 `TileWeapon = 'W'`）
2. 在 `Game` 结构体加对应 HashMap（如 `Weapons map[[2]int]Weapon`）
3. `mapgen.go` 的 `PlaceEntities` 增加放置逻辑
4. `battle.go` 的 `MovePlayer` 增加拾取效果分支
5. `render.go` 的 `Render` 构建 screen 时把 HashMap 画进去
6. `mapgen.go` 的 `hasEntity` 增加占位检测

### 7.3 实现怪物 AI

在 `battle.go` 的 `MovePlayer` 末尾（玩家移动完成、触发格子效果之后）加一段怪物 AI tick：遍历所有怪物，朝玩家方向走一格（用寻路或简单贪心），并检测是否与玩家相邻——相邻则触发战斗。注意：怪物移动后也要调用 `UpdateVisibility()`，因为玩家视野范围是相对坐标的，怪物移动不影响视野但会影响下一次渲染。

### 7.4 增加关卡特性

「每层特殊规则」可以在 `Game` 结构体加 `FloorEffects` 字段，在 `NextFloor()` 时随机抽取，在 `MovePlayer` / `BattleAttack` 的对应位置加效果判断。例如：黑暗层（视野半径临时改为 3）、毒雾层（每回合 −1 HP）、宝藏层（金币数量 ×3）。

### 7.5 终端兼容性

Windows 7 及更早系统不支持 ENABLE_VIRTUAL_TERMINAL_PROCESSING。如果需要兼容，可在 `enableWindowsANSI()` 里检测返回值，失败则 fallback 到纯黑白渲染（不插入 `\x1b[90m` / `\x1b[0m`）。Linux/macOS 终端默认支持 ANSI，无需处理。

## 8. 已知的简化与取舍

| 取舍 | 说明 |
|------|------|
| 怪物完全被动 | 只有玩家撞到才触发战斗，没有巡逻/追击。简化了回合同步的复杂度，但少了紧张感。 |
| 无障碍物视野 | 视野是纯圆形，不被墙壁遮挡（即你可以穿墙看到隔壁房间）。真正的视线遮挡需要实现 Bresenham 射线或阴影投射算法，但 60×20 的小地图上穿墙视野影响不大，且实现成本高。 |
| 地图不可破坏 | 墙壁是静态的，没有挖掘/爆炸机制。 |
| 单条战斗日志 | Battle.Log 不限长度，会一直 append；战斗结束后 Battle 被置 nil 自动释放，所以不会泄漏。如需滚动显示限制，可限制最近 N 条。 |
| stdin 原始模式 | `readKey()` 直接读 stdin，要求终端在外部已经设置为 raw 模式（按一次键立即回读，不需要回车）。Windows cmd/PowerShell 默认是 cooked 模式，需要回车确认。如需真正的即时按键，应调用 `SetConsoleMode` 关闭 `ENABLE_LINE_INPUT` + `ENABLE_ECHO_INPUT`。 |
